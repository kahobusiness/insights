## 二、推出每 token 成本公式

### 钱花在哪？

运营推理服务，你付的是**「租 GPU 的时间费」**。比如 H100 大约 $2/小时。所以一段时间内的总成本：

$$
\text{总成本} = \text{GPU 单价} \times \text{这段时间的长度}
$$

为简化，我们把 GPU 单价当作 1（选一个货币单位）。那么一次推理的总成本就直接等于 `T`：

$$
\text{某次推理的总成本} = T
$$

### 单位：每 token 成本

但运营方关心的不是「一次推理多少钱」，而是「**每个 token 多少钱**」——因为他向用户也是按 token 收费。

那这次推理产出了多少 token？

> 🔑 **关键洞察**：一次「前向传播」（把所有参数读一遍 + 算一遍），**生成了 B 个 token**——batch 里每个用户各拿到一个新 token。
> 

所以：

$$
\text{每 token 成本} = \frac{\text{总成本}}{\text{产出的 token 数}} = \frac{T}{B}
$$

> 🔑 **这就是访谈里 GPU 工程师在白板上画的那个 `T/B`**。Y 轴就是它。
> 

---

## 三、把 T/B 写出来：可摊销 vs 不可摊销

回想上一篇的核心方程：

$$
T = \max(T_{memory},\ T_{compute})
$$

$$
T_{memory} = \frac{N_{total}}{BW} + \frac{B \times L \times b}{BW}, \quad T_{compute} = \frac{B \times N_{active}}{C}
$$

每 token 成本 = `T/B`，把每一项都除以 B，神奇的事情发生了：

### T_compute / B：算力常数

$$
\frac{T_{compute}}{B} = \frac{B \times N_{active} / C}{B} = \frac{N_{active}}{C}
$$

> 🔑 **算力成本是常数**——不依赖 B。这很合理：每个用户都要「算自己那份」，所以平均到每用户的算力时间就是固定的。
> 

### T_memory / B：双曲线 + 常数

$$
\frac{T_{memory}}{B} = \underbrace{\frac{N_{total}}{B \times BW}}_{\text{双曲线}} + \underbrace{\frac{L \times b}{BW}}_{\text{常数}}
$$

> 🔑 **两项形态完全不同：**
> - **搬参数项除以 B 之后变成双曲线**——B 越大这一项越小，**这就是 batch 的经济学价值**！
> - **搬 KV 项除以 B 之后变成常数**——因为 B 既出现在分子（用户多就要搬多）又出现在分母（摊给多人），完全抵消了
> 

后者正是我们一直说的「**KV 摊不掉**」——不管 batch 多大，人均 KV 搬运成本不变。

### 合起来：每 token 成本

<Callout type="info" emoji="🎯">

$$
\text{Cost/token} = \max\left(\underbrace{\frac{N_{total}}{B \times BW}}_{\text{搬参数（双曲线）}} + \underbrace{\frac{L \times b}{BW}}_{\text{搬 KV（常数）}},\ \underbrace{\frac{N_{active}}{C}}_{\text{算力（常数）}}\right)
$$

三项中只有第一项被 B 摊销，后两项都摊不掉。

</Callout>

## 四、画出成本图

横轴还是 B，但 Y 轴是 **每 token 成本**。我们一项一项画。

### 第一项：搬参数 / B（双曲线）

`N_total / (B × BW)`，B 越大越小：

```
成本
  │
  │ ╲
  │  ╲
  │   ╲
  │    ╲
  │      ╲
  │        ╲___
  │            ─────────────
  │
  └─────────────────────► B
  0
```

> 🔑 **B 趋近 0 时这一项趋于无穷大**——一个用户独占所有参数搬运成本，完全没摊销，贵到离谱。这就是为什么「不 batch」的经济学根本不成立。
> 

### 第二项：搬 KV / B（常数，KV 线）

`L × b / BW`，水平线，高度由上下文长度决定。

### 第三项：算力 / B（常数，算力线）

`N_active / C`，另一条水平线。

### 三项叠加：要小心几何关系

把三项叠加，**关键问题**：算力线和 KV 线，谁高谁低？

要让甜蜜点（双曲线下降时与水平线相交）能存在，**算力常数线必须高于 KV 常数线**：

$$
\frac{N_{active}}{C} > \frac{L \times b}{BW}
$$

> 🔑 **物理含义**：在「正常」的模型 + 上下文配置下，算力是比 KV 更「贵」的成本——所以达到甜蜜点之后，瓶颈是算力，不是 KV。这跟上一篇默认的「短上下文场景内存项可忽略」是一致的。
> 

修正后的成本图（**正确版本**）是这样：

```
成本
  │ ╲
  │  ╲           ← 双曲线段（memory-bound）
  │   ╲             双曲线 + KV 常数
  │    ╲
  │     ╲
  │      ╲___
  │  ─────────╳───────────── ← 算力常数 N_active/C
  │           ↑ 甜蜜点
  │
  │  ─────────────────────  ← KV 常数 L×b/BW（在算力常数下方）
  │
  └──────────────────────► B
```

最终成本曲线 = 取上面那条：

```
成本
  │ ╲
  │  ╲      ← memory-bound 区（双曲线主导）
  │   ╲
  │    ╲
  │     ╲___
  │  ────────╳──────────── ← compute-bound 区（算力常数主导）
  │          ↑
  │        甜蜜点
  │
  └──────────────────────► B
  0
```

## 五、读这张图：三个区域

| 区域 | 谁主导 | 状态 | 运营方处境 |
| --- | --- | --- | --- |
| **B 很小** | 双曲线 | 成本爆炸 | 运营方亏死 |
| **甜蜜点** | 双曲线 = 算力 | 成本接近最低点 | 最佳运营点 |
| **B 很大** | 算力常数 | 成本平稳在算力下界 | 饱和，再加 batch 没收益 |

### 成本下界

注意右侧的水平渐近线高度。在正常配置下（算力线高于 KV 线）：

$$
\text{Cost/token 下界} \approx \frac{N_{active}}{C}
$$

> 🔑 **这是单 token 成本的物理下限**。它的意思是「工人全速干活时，每个用户那份工作的成本」——再大的 batch 也省不了，因为算的工作量是每用户独有的，batch 帮不上忙。
>

## 六、回答三个常见问题

有了这张图，前两篇里反复出现的几个问题都能给出干净的几何答案。

### 问题 1：「不 batch 差 1000 倍」从哪来？

代入 `B=1` 时（一个用户独占）的成本：

$$
\text{Cost/token}(B=1) \approx \frac{N_{total}}{BW}
$$

代入甜蜜点（B ≈ 几千）时的成本：

$$
\text{Cost/token}(\text{甜蜜点}) \approx \frac{N_{active}}{C}
$$

两者比例：

$$
\frac{N_{total}/BW}{N_{active}/C} = \frac{N_{total}}{N_{active}} \times \frac{C}{BW} = \text{稀疏度倒数} \times 300
$$

代入 DeepSeek V3：`18 × 300 ≈ 5400 倍`。

> 🔑 **这就是「差 1000 倍」的精确来源——其实是 5400 倍**。不 batch 的经济学根本不成立，这不是工程偷懒，是物理硬性。
> 

### 问题 2：为什么 AI 服务有强烈的规模经济？

把成本曲线想象成一个梯子：

- **大厂（OpenAI / Anthropic）**：用户量充足，永远把 batch 钉在甜蜜点 → 人均成本接近下界
- **小厂**：用户量不到甜蜜点，比如 batch 只能撑到 500，最优是 5000 → **卡在双曲线下降段中部**
- **独占模式**：单个企业客户独占 GPU → 卡在最左端，成本指数级高

> 🔑 **「用户量增长直接转化为利润率改善」是 AI 服务的结构性性质**——只要还在双曲线段，每多一个用户都是边际成本递减。这也是为什么大厂对小厂有结构性成本优势——**小厂 batch 开不起来，人均成本永远高一档**。
> 

### 问题 3：OpenRouter 这类聚合层的商业逻辑是什么？

> 📌 **本质**：把多个小客户的请求合并成一个大 batch——把每个客户单独跑达不到的 batch，聚合到能达到甜蜜点的 batch。它的毛利就来自这个 batch 摊销的差价。
> 

注意聚合本身有天花板：**不同模型不能共享 batch**，所以一个聚合层的有效 batch 仍然受限于它服务的模型分布。

## 七、回到 FastMode 与 SlowMode 之谜

第一篇开头我留了一个问题：能不能「100× 价格换 100× 速度」、或者「100× 慢换 100× 便宜」？现在你能自己用图回答了。

### 把延迟图和成本图叠在一起

```
延迟图（用户视角）        成本图（运营视角）

T                          Cost/token
│      ╱                  │ ╲
│    ╱                    │  ╲
│  ╳   ← 延迟最优在交叉点  │   ╲___
│ ╱                        │      ╳ ← 成本最优也在交叉点
│                           │       ─────── 之后水平
│                           │
└──────► B                  └──────────► B
   ↑                            ↑
  在交叉点之前：               在交叉点之前：
  加 batch 不影响延迟         加 batch 显著降成本

  在交叉点之后：               在交叉点之后：
  加 batch 直接增加延迟        加 batch 几乎不再降成本
```

> 🔑 **几何关键事实**：**两张图的交叉点是同一个 B**——都是 `B = 300 × 稀疏度`，因为底层是同一个方程。
> 

### 这意味着什么？

运营方面对一个抉择：

- **B = 甜蜜点**：成本接近下界 ✅，延迟也接近下界 ✅，**完美**
- **B > 甜蜜点**：成本几乎不再降低，但延迟开始线性上涨
- **B < 甜蜜点**：延迟没变好（还是物理下界），成本指数级上涨，**纯亏**

> 🔑 **理性运营点就是甜蜜点**。不是「在用户体验和成本之间艰难权衡」——**两者在同一点同时最优**。这是 LLM 推理经济学最优雅的一个性质。
> 

### FastMode 的本质

**FastMode**（比如 6× 价格换 2.5× 速度）做的事：让你独占 batch 中更大的份额，把整个系统从甜蜜点拉到「B 更小」的地方。

- 普通模式：你和 5000 人共享一台机器
- FastMode：让你「占 6 个用户的位置」，等效 batch 变小 → 移到延迟图的更左边
- 但延迟图左边是平的（物理下界），所以**速度只能提升有限的倍数**
- 同时把成本曲线推到双曲线段，成本爆涨 → 所以收 6× 价格

> 🔑 **为什么不能「100× 价格换 100× 速度」？** 因为延迟下界是参数搬运时间这个**物理墙**。FastMode 能做的最多就是把你从 compute-bound 区拉回 memory-bound 区，达到那个延迟下界——再多花钱也撞不破这堵墙。
> 

### 那「便宜+慢」呢？

也讲不通。因为成本曲线在 `B > 甜蜜点` 之后是**水平的**——你愿意多等也省不了钱。

> 🔑 **结论**：
> - ✅ 「贵+快」可行（直到撞墙）
> - ❌ 「便宜+慢」不可行（成本下界是硬性物理常数）
>

## 八、一张表收尾

| 量 | 公式 | 几何 | 对 B 的依赖 |
| --- | --- | --- | --- |
| 搬参数成本 / token | `N_total / (B × BW)` | 双曲线 | **可摊销**（B 越大越便宜） |
| 搬 KV 成本 / token | `L × b / BW` | 水平线 | **摊不掉** |
| 算力成本 / token | `N_active / C` | 水平线 | **摊不掉** |
| 总成本 / token | `max(双曲+KV, 算力)` | 先降后平 | 在甜蜜点最低 |
| 成本下界 | `≈ N_active / C` | 右侧水平渐近线 | B 趋于无穷时的极限 |

核心方程：

$$
\boxed{\ \text{Cost/token} = \max\left(\frac{N_{total}}{B \cdot BW} + \frac{L \cdot b}{BW},\ \frac{N_{active}}{C}\right)\ }
$$

「不 batch 差多少倍」的来源：

$$
\boxed{\ \frac{\text{Cost}(B=1)}{\text{Cost}(\text{甜蜜点})} \approx 300 \times \frac{N_{total}}{N_{active}}\ }
$$
